## 项目背景

BI 平台面对的是分析师、运营和业务同学的自助分析需求。用户不应该每次想看“某类用户经过哪些行为路径”“某个标签命中的用户有哪些”“某个分析结果能不能保存成分群”时都去找研发写 SQL。真正要解决的是：把业务侧的分析配置翻译成稳定、可控、可复用的查询计划。

我参与的数数 BI 平台，重点是在后端把复杂分析请求转换成可执行、可维护、可扩展的 SQL 计划，并把计算结果稳定地交给上层页面使用。

## 我的职责

- 参与用户标签、用户分群、用户行为序列、路径分析、属性分析、下钻查询、保存结果为分群和公共定时任务等模块。
- 在路径分析中处理请求校验、时间范围归一化、事件候选集、过滤条件、虚拟事件、虚拟属性、维度属性、用户标签和用户分群等语义。
- 负责 StarRocks 查询链路中的 SQL 规划、CTE 组装、结果聚合和下钻结果组装。
- 在用户分群计算中处理异步计算、计算日志、CSV ID 校验、结果表替换、下钻保存结果分群和依赖刷新等能力。

## 技术方案

BI 后端的难点不是拼一个 SQL，而是把“页面上的分析配置”稳定翻译成数据库能理解的查询计划。路径分析里，我参与的 SQL Builder 会先做请求级校验和时间归一化，再解析事件、过滤、拆分字段、虚拟属性、维度属性、用户标签和用户分群，最后生成包含 `base_events`、`ordered_events`、`sessionized`、`anchored`、`path_rows`、`aggregated_paths` 的 CTE SQL。

在路径展开上，系统使用 `LAG` 判断同一用户相邻事件间隔，通过累计 sessionStartFlag 得到会话编号，再以源事件为锚点，用 `LEAD/LAG` 展开固定 10 步路径。这个设计让路径分析既能支持正向路径，也能支持反向路径，同时避免在 StarRocks 中构造过长的 session 字符串。

在分群计算上，系统不会直接覆盖正式结果表，而是先写临时表，写入成功后再创建正式表并执行表交换，最后清理临时表。对于“下钻结果保存为分群”的场景，还会先探测 SQL 元数据，确保下钻 SQL 只返回一列主体值；再去掉页面预览用的最外层 LIMIT，保存完整主体集合。这能避免把“预览 1000 条”误当成真实分群结果。

## 关键优化

- **查询安全阈值**：路径分析设置候选事件数量和过滤条件数量上限，避免用户配置过大导致 SQL 失控。
- **语义复用**：虚拟事件、虚拟属性、维度属性、用户标签和用户分群统一进入分析语义层，减少不同分析模块重复实现同类解析逻辑。
- **结果一致性**：用户分群计算通过临时表写入后再 swap 正式结果表，减少中间状态对线上查询的影响。
- **分析准确性**：保存结果分群时去掉预览 LIMIT，并将主体值统一转换为字符串列，避免保存到不完整或列结构错误的数据。
- **排障效率**：计算日志、执行 SQL 打印和任务状态沉淀，让问题定位从“猜是哪段 SQL 坏了”变成有上下文可追踪。

## 业务成效

这个项目的价值，是把原本依赖研发写 SQL 的分析需求，转成业务同学可以配置的自助分析能力。常见标签查看、分群圈选、路径分析和结果下钻，不再需要每次排研发需求；典型取数和验证场景可以从天级等待缩短到小时级甚至分钟级。

更重要的是，它不是一次性 SQL 报表，而是一套可继续扩展的分析后端：新增分析类型、下钻能力或保存结果能力时，可以复用语义准备、SQL 规划、执行、日志和结果管理链路。

从个人能力上，它体现了我对分析型系统的理解：既要懂业务语义，也要懂 SQL 执行、数据一致性、异步任务和性能边界。
