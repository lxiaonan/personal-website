幂等不是给接口加一个 token 就结束了。它更像一组约束：同一业务动作无论被触发多少次，最终都应该得到一致的业务结果。

## 请求唯一性

先定义什么叫“同一次业务动作”。常见做法包括：

- 客户端生成请求号。
- 服务端生成业务单号。
- 用外部支付单号、订单号或任务号作为唯一键。

关键点是唯一标识必须进入持久化层，不能只存在于缓存。

## 状态机约束

幂等经常和状态流转绑定。比如订单只能从 `CREATED` 变成 `PAID`，不能从 `CLOSED` 重新变成 `PAID`。

数据库更新语句里可以显式带上当前状态：

```sql
UPDATE orders
SET status = 'PAID'
WHERE id = ? AND status = 'CREATED';
```

如果影响行数为 0，就要根据当前状态判断是重复请求、非法请求还是数据异常。

## 数据库唯一约束

唯一索引是幂等设计里最可靠的防线之一。缓存可以优化性能，但数据库约束才是最后的裁判。

例如支付回调可以对 `payment_no` 建唯一索引，重复回调时捕获唯一键冲突，再查询已有结果返回。

## 异步补偿

幂等接口通常还需要补偿任务。网络超时、MQ 重投、第三方回调重复都会让链路变得不确定。

建议把关键动作记录成可查询的业务日志或任务表，并保证补偿任务也走同一套幂等入口。
