一个 Java 后端程序员的个人网站应该放什么

个人网站第一版不需要复杂后台，先把身份、项目、文章和联系方式讲清楚。

  • Personal Site
  • Career
  • Java

个人网站的第一目标不是炫技，而是降低别人理解你的成本。

对 Java 后端开发者来说，一个好用的网站至少回答四个问题：

  1. 你主要做什么方向。
  2. 你解决过哪些真实问题。
  3. 你的代码、文章和项目在哪里。
  4. 别人如何联系你。

首页先讲能力

首页应该像一个清晰的工程师 Profile，而不是文章列表。技术栈可以写，但不要只堆名词。更好的写法是把技术栈和问题绑定：

  • 用 Redis 做缓存、限流、分布式锁时遇到过什么边界。
  • 用 MQ 处理异步链路时如何保证幂等和可观测。
  • 用 MySQL 时如何定位慢查询和设计索引。

文章用来证明判断力

技术文章不必追求数量。更值得写的是那些你真的踩过、想清楚过的问题，比如缓存一致性、接口幂等、数据库事务、部署回滚、线上事故复盘。

个人站最好先用 Markdown 或 MDX 管理内容。Git 提交历史天然就是文章版本记录，也方便以后迁移。

项目写出上下文

项目页不要只写仓库链接。每个项目都应该写清楚：

  • 背景是什么。
  • 你负责哪一部分。
  • 关键技术决策是什么。
  • 最后带来了什么结果。

这比单纯罗列 Spring Boot、MySQL、Redis 更有说服力。

一个更贴近后端开发者的展示方式

如果个人网站只是把技术栈写成一排标签，别人很难判断你到底会不会解决真实问题。比如写“熟悉 Spring Boot、Redis、MySQL、MQ”，这句话几乎每个后端简历都能写；但如果换成“在广告投放平台里处理过多媒体 API 拉取、批量创建、素材关系同步和任务补偿”，信息密度就完全不同。因为它说明你接触过外部系统、异步任务、数据一致性、失败重试和业务规则，不只是会写接口。

我更推荐把网站当成一个工程师的证据链：主页讲定位，项目页讲真实系统，文章页讲思考过程。比如广告平台可以拆成“多媒体渠道接入”“批量创建投放计划”“花费数据定时拉取”“素材与创意关系维护”；BI 平台可以拆成“路径分析 SQL 生成”“用户标签历史版本”“结果分群保存”“属性分析下钻”。这些模块不一定都要开源，但可以写清楚你解决了什么问题、为什么这么设计、边界在哪里。

项目介绍也不要只写“负责后端开发”。更好的写法是：这个需求原来靠人工在媒体后台配置，容易漏字段、命名不统一、失败后难以追踪；后端把它抽象成模板、预览、提交、结果查询和失败重提几个阶段，减少重复配置，也降低错误提交到媒体端的概率。这样读者能看到你的业务理解、抽象能力和风险意识。

文章同样要服务展示目标。个人网站里的文章不需要像教程平台一样从零讲概念，而应该更像“我在实际开发里怎么处理这个问题”。比如写缓存，不要只讲 Redis 五种数据结构，可以写“运营配置变更后，列表页缓存如何失效，删除失败如何补偿，如何观察缓存命中率”。写幂等，不要只讲 token，可以写“批量创建任务被重复提交时，如何用批次号、业务唯一键和状态机保证不会重复创建”。这种文章虽然不一定最长，但更有辨识度。

最后，个人网站第一版不应该追求复杂后台。Astro + Markdown 已经足够表达项目、文章和链接，部署到 GitHub Pages 也能让维护成本很低。真正应该投入时间的是内容结构：标题是否像人写的，项目是否有上下文，文章是否能看出你做过真实系统。网站的价值不是页面多炫，而是让别人用更短时间判断：这个人能把复杂业务拆清楚，并且能把后端链路做稳。
